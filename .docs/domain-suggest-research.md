# Deep Research: Skills для подбора домена под стартап / новую компанию / проект

Дата: 2026-06-20

## Краткий вывод

Рынок AI-инструментов для доменов резко асимметричен:

- **Проверка доступности** реализована везде - десятки MCP-серверов и CLI (RDAP/WHOIS/DNS).
- **Генерация/подбор имён** почти не покрыта в слое skills, а единственный публичный Claude-skill (`domain-name-brainstormer`) **галлюцинирует доступность и цены**, потому что не привязан ни к какому реальному checker'у.
- **Связки "генерация + реальная проверка"** в одном инструменте - большая редкость (единичный пример: `domain-finder-mcp`, и тот требует внешних LLM-ключей + Namecheap/Domainr).

Вывод для этого репозитория: правильный следующий шаг - отдельный **`domain-suggest`** skill, который генерирует кандидатов (нейминг-методология внутри LLM, без внешнего API) и **прогоняет их через уже существующий `domain-check`** для настоящей проверки. Это закрывает ровно тот разрыв, который в этой сессии закрывался руками: брейншторм -> проверка -> добор вариантов.

## Контекст репозитория

В `ai-skills` уже есть два доменных skill, и **оба только проверяют**, а генерацию явно исключают:

- `domain-check` - no-API RDAP/WHOIS/DNS (`SKILL.md`: "Do not use for generating domain name suggestions"). После доработок этой сессии: concurrency, streaming, кэш bootstrap, retry.
- `regru` - проверка через REG.RU API 2.

`TASKS.md` (T2) прямо фиксирует: "Do not trigger for domain name brainstorming unless exact domains are supplied for checking." То есть генерация сознательно вынесена за скоуп checker'а - и логично оформить её как **отдельный** skill, а не расширять `domain-check`.

## Разбор прецедента: `domain-name-brainstormer`

Это де-факто эталон "как выглядит наминг-skill" в формате SKILL.md. Что внутри:
- frontmatter description: "Generates creative domain name ideas ... and checks availability across multiple TLDs".
- инструкции агенту: понять проект -> сгенерировать имена -> "проверить" доступность -> дать альтернативы -> объяснить брендинг.
- встроенные эвристики: <=15 символов, произносимость, без дефисов/цифр, гайд по TLD.

**Критический изъян:** в skill **нет ни одного скрипта и ни одной привязки к tool/MCP**. В примере вывода стоят галочки доступности и цены ($2,500, premium) - всё это **выдумывается LLM**. Реальная проверка появляется только если параллельно подключить доменный MCP.

Именно здесь у вас структурное преимущество: `domain-check` даёт настоящую проверку через RDAP/WHOIS без API-ключей. Связав генерацию с ним, получаем то, чего у прецедента нет.

## Эталон архитектуры: `domain-finder-mcp`

Лучший по дизайну инструмент "генерация+проверка". Три tool'а:
- `suggest-domains` - входы: `businessDescription`, `searchMode` (standard/competitive/budget/premium/international), `industry`, `maxSuggestions` (1-50).
- `check-domain` - один домен или батч <=20.
- `deep-tld` - перебор 1441+ TLD батчами, `creativityLevel` (conservative/moderate/wild), опциональная проверка.

Минусы для нашего кейса: нужен внешний LLM (Ollama/OpenAI/Groq/Together) и платный доступ Namecheap (20+ доменов, $50 баланс) либо Domainr. У нас LLM - это сам агент, а проверка уже есть бесплатно. То есть мы можем повторить лучшие идеи (режимы поиска, уровни креативности, deep-TLD), но без внешних зависимостей.

## Таксономия техник генерации (с примерами)

Сводка приёмов, которыми пользуются коммерческие генераторы (Namelix, Lean Domain Search, Namecheap Beast Mode, NameMesh, BustAName, Wordoid и др.):

1. **Аффиксация** - приставки/суффиксы к ключу: `get-/go-/my-/try-`, `-ly/-ify/-hub/-base/-kit/-labs/-io` -> Shopify, Grammarly, GitHub.
2. **Компаундинг** - склейка двух реальных слов -> Facebook, Snowflake, Dropbox.
3. **Блендинг/портманто** - слияние по общему звуку -> Pinterest (pin+interest), Instagram, Groupon.
4. **Фонетическая замена/респеллинг** - c->k, s->z, выпадение гласных -> Flickr, Lyft, Kasa.
5. **Семантическое расширение** - синонимы/тезаурус: "fast" -> swift, bolt, dash; "money" -> mint, vault, ledger.
6. **Sounds-like / рифмы** - близкие омофоны и рифмопары.
7. **Абстрактные/коины** - придуманные произносимые слова, лат./греч. корни -> Kodak, Spotify, Verizon.
8. **Слоговый/Markov-синтез** - статистически "словоподобные" токены (Wordoid).
9. **AI/LLM-семантика** - из описания/вайба в он-бренд кандидатов -> "calm meditation app" -> Stillwave, Lumora.
10. **Курирование/крауд** - ручной отбор премиум-имён (BrandBucket принимает <10% заявок).

**Спец-приём из этой сессии (стоит зашить как отдельную технику):** *спрятать сигнал в слове* - буквы целевого концепта вшиты в реальное слово. Примеры из работы по neira: `avail**a**` (available, прячет "ai"), `s**ai**l`~sale, `cl**ai**m`, `g**ai**rd`~guard. Это гибрид блендинга и фонетической игры, который реальные генераторы почти не умеют.

## Методология нейминга (что зашить в инструкции skill)

### Типы имён
| Тип | Суть | Примеры |
|---|---|---|
| Описательное | прямо говорит, что делает | PayPal, General Motors |
| Suggestive/эвокативное | намекает на пользу/чувство (sweet spot) | Salesforce, Slack, Stripe |
| Коин/неологизм | придуманное, максимально защитимо | Kodak, Spotify, Verizon |
| Абстрактное | реальное слово без связи с продуктом | Apple, Amazon, Oracle |
| Акроним | из инициалов (слабое без бюджета) | IBM, IKEA |
| Эпонимическое | по фамилии основателя | Tesla, Dell |
| Метафора | образ -> качества | Nike, Amazon (река) |
| Компаунд/портманто | склейка/слияние | Facebook, Netflix |
| Иностранное слово | смысл/звук из другого языка | Audi, Volvo |

Консенсус индустрии (Igor, Lexicon, A16Z): **suggestive / коин / абстрактное** - самые защитимые и брендовые; **описательное / акроним** - самые слабые вдолгую.

### Тесты качества (зашить как чек-лист)
- **SMILE** (Alexandra Watkins): **S**uggestive, **M**emorable, **I**magery, **L**egs, **E**motional.
- **SCRATCH** (чего избегать): **S**pelling-challenged, **C**opycat, **R**estrictive, **A**nnoying, **T**ame, **C**urse of knowledge, **H**ard to pronounce.
- Радио-тест (запиши на слух без ошибок), <=3 слогов, без дефисов/цифр, нет негатива на ключевых языках, запас на пивот.

### Звуковой символизм (для брендируемости)
- Передние гласные /i/ ("ee") -> маленькое/быстрое/острое; задние /a/,/o/,/u/ -> крупное/тяжёлое.
- Взрывные /k/,/t/,/p/ -> резкое/энергичное (tech/edge); сонорные /m/,/l/,/b/ -> мягкое/дружелюбное (care/comfort).
- Полезно для коинов: фонестемы (gl- -> свет: gleam, glow).

### TLD-стратегия
- **.com** - дефолт доверия (Paul Graham: нет x.com -> меняй имя). Для масс-/enterprise - приоритет.
- **.ai** - мейнстрим для AI-стартапов (Google трактует как generic), но дорогой и часто 2-летний минимум.
- **.io/.co** - ок для tech/dev-аудитории; .co - слабее, риск утечки на .com.
- **ccTLD/.ru/.рф** - для регионального фокуса (прямой кейс этого репозитория).
- Правило: экзакт-`.com` максимизирует доверие; **коин-имена** делают экзакт-мэтч реально достижимым.

### Юр/практический чек-лист
- Trademark: USPTO TESS (US), WIPO Global Brand DB, EUIPO - обязательно по нужному Nice-классу.
- Соц-хэндлы (X/IG/LinkedIn/GitHub), app store, SEO-конкуренция дженерик-слов.

## Рекомендация: новый skill `domain-suggest`

**Назначение:** из описания проекта/ключей/вайба/ограничений сгенерировать ранжированный список доменов-кандидатов и **сразу проверить реальную доступность** через `domain-check`, затем догенерировать замены для занятых.

**Почему отдельный skill, а не расширение `domain-check`:** checker сознательно read-only и exact-only (см. TASKS.md T2). Генерация - другой режим работы и другой триггер. Разделение сохраняет чистоту роутинга и описаний.

**Ключевое преимущество перед `domain-name-brainstormer`:** генерацию делает сам агент (LLM), а доступность даёт **детерминированный скрипт**, а не галлюцинация. Никаких внешних LLM/registrar-ключей.

**Предлагаемая структура:**

```text
domain-suggest/
├── SKILL.md
├── references/
│   ├── naming-methodology.md
│   └── generation-techniques.md
└── evals/
    └── evals.json
```

Скрипт не обязателен: генерация - задача LLM; проверка делегируется `domain-check/scripts/domain-check.mjs`. Опционально - тонкий wrapper, который принимает список и зовёт `domain-check`.

**Входы (из промпта):** описание проекта/индустрия; ключевые слова (включить/исключить); вайб/тон; целевая длина; язык(и); приоритетные зоны (.ru/.ai/.com...); ограничения (без "bot", без транслитерации и т.п. - ровно как в этой сессии).

**Выход:** сгруппированный по зонам ранжированный список со статусом доступности и краткой причиной ("почему имя работает"), топ-пик + раннер-ап, и следующий шаг (добор замен / регистрация / проверка trademark+хэндлов).

**Рабочий цикл (зашить в SKILL.md):**
1. Снять/уточнить бриф (1 короткий вопрос, если критично не хватает).
2. Сгенерировать N кандидатов разными техниками (аффиксация, блендинг, коины, wordplay...), отфильтровать через SMILE/SCRATCH и ограничения.
3. Прогнать батч через `domain-check` (параллельно, с retry - уже поддержано).
4. Для занятых - доген заменами, повторить проверку.
5. Выдать ранжированный список + рекомендации, явно отметив "доступность по RDAP/WHOIS, регистрация не гарантирована" (правило безопасности из `domain-check`).

**Границы (description в SKILL.md):**
- Триггерить: "придумай/подбери домен/название для...", "brainstorm domain names...", "нужно имя для стартапа/проекта".
- Не триггерить: точечная проверка конкретных доменов (-> `domain-check`), регистрация/трансфер/DNS (-> `regru`/вне скоупа).

**Что осознанно НЕ делаем в v1:** реальная проверка trademark и соц-хэндлов (выдаём как рекомендованные ручные шаги со ссылками USPTO/WIPO) - чтобы не раздувать скоуп и не плодить хрупкие интеграции.
